Building fully local LLM inference on Android with llama.cpp

A short project note on getting a local Android chatbot running with GGUF models, Kotlin UI code, and a small retrieval experiment for grounded answers.

What I built

The end result is a simple Android app that runs a GGUF model entirely on-device. The app imports a model file, copies it into app-private storage, loads it through the Android llama.cpp binding, and answers prompts without relying on a cloud backend.

Integration path

I initially tried integrating the Android llama.cpp pieces directly into a custom app. That path exposed the hard part of on-device AI on Android: the UI is easy, but native packaging is not. After repeated CMake, ABI, and backend-loading issues, I moved to the official examples/llama.android project as the base integration and then customized the app layer from there.

How the app works

The runtime flow is straightforward once the native setup is correct:

  1. Select a GGUF file through Android's document picker.
  2. Parse GGUF metadata to inspect the selected model.
  3. Copy the model into filesDir/models.
  4. Initialize the inference engine through AiChat.getInferenceEngine(...).
  5. Load the model with engine.loadModel(modelFile.path).
  6. Send a prompt and collect generated tokens through Kotlin Flow.

Kotlin app layer

Kotlin handled the user-facing flow: selecting a GGUF file, copying it into app-private storage, initializing the inference engine, sending prompts, and rendering streaming output in the UI. The native model runtime stayed inside the llama.cpp Android binding.

Retrieval experiment

I also explored a lightweight retrieval flow for source-grounded answers. The first pass used simple query embedding and matching experiments. For a leaner proof of concept, I later tested a lightweight in-memory retrieval layer and compared prompt behavior across small GGUF instruct models. The main lesson was that retrieval can work while weaker models still fail to answer grounded questions well. Model quality matters a lot.

Build issues encountered

Most of the work was not in the UI. The hard issues were native and packaging related:

  • CMake path and add_subdirectory mistakes while integrating llama.cpp into a separate Android project.
  • Gradle plugin and version-catalog mismatches when trying to reuse the sample modules elsewhere.
  • Android ABI mismatches between emulator/device and packaged native libraries.
  • Minimum SDK and AndroidX configuration mismatches across modules.
  • Backend loading failures such as native runtime errors where no GGML backend was available at model load time.
  • Instruction-following differences across smaller GGUF models during the retrieval experiment.

Why the official sample mattered

The official Android sample already solves the hard native integration layer. Reusing that base was the pragmatic move because it let me focus on application logic instead of fighting backend packaging issues repeatedly.

How to run

  1. Open examples/llama.android in Android Studio.
  2. Sync Gradle and build the project.
  3. Run the app on a compatible Android device or emulator.
  4. Pick a GGUF model from storage.
  5. Wait for the model to copy into app-private storage and load.
  6. Send prompts and observe local inference output.

How to debug

The fastest debugging path was Logcat. The most useful signals were:

  • Device ABI logs to confirm the app was running under the expected architecture.
  • Model import path logs to verify the GGUF was copied into private storage correctly.
  • Native engine logs during loadModel() to spot backend-loading failures.
  • Prompt and retrieval debug logs to verify whether context was actually being passed to the model.
  • Token stream logs to distinguish between model failure and retrieval failure.

Useful checks

adb logcat | grep -E "ABI|RAG_|InferenceEngineImpl|ai-chat"
adb shell ls -l /data/user/0/<package-name>/files/models
unzip -l app-debug.apk | grep "lib/"

Main lesson

On-device AI is not only about loading a model. Native packaging, ABI compatibility, backend loading, prompt structure, and model choice all matter. Once the integration foundation is stable, the Kotlin app layer becomes straightforward.
